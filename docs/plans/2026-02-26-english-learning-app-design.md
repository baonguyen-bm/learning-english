# Design Document: Personalized English Learning App (PWA)

## 1. Overview

A personalized English learning Progressive Web App focused on improving pronunciation, listening-to-spelling skills, and workplace communication. The app provides a structured "Daily Mission" (~15-20 mins) comprising dictation, spelling, and speaking exercises themed around real workplace scenarios.

**Core Philosophy:** "Hear it, Write it, Say it."
**Target User:** Busy professional with 15-20 minutes/day for focused self-study.
**Key Constraint:** Zero-cost runtime. No paid API keys. Content is pre-generated and seeded into the database.

## 2. Architecture & Tech Stack

### Frontend (The App)
* **Framework:** Next.js 14+ (App Router) — easy Vercel deployment, PWA capabilities, React ecosystem.
* **Styling:** Tailwind CSS — rapid UI development, mobile-responsive.
* **State Management:** React Context / Hooks (sufficient for this scale).
* **PWA:** `@ducanh2912/next-pwa` (maintained fork) for installable app shell and static asset caching.

### Backend (The Data)
* **Database:** Supabase (PostgreSQL) — excellent free tier, built-in Auth.
* **Authentication:** Supabase Auth (Email/Password or Magic Link).

### Audio & Speech (Browser-Native)
* **Text-to-Speech (TTS):** `window.speechSynthesis` — reads sentences and words aloud. Free, works offline.
* **Speech-to-Text (STT):** `window.webkitSpeechRecognition` — transcribes user speech for pronunciation validation. Free, requires online on some browsers (Chrome sends audio to Google servers; Safari processes locally).

### Content Strategy
Content (dialogs, vocabulary, sentences) is pre-generated by the AI Assistant during development. Stored as structured JSONB in the `missions` table. Each mission is themed around a workplace scenario (e.g., "Daily Standup", "Client Email", "Code Review"). The app requires no external AI API keys at runtime.

## 2.5 Design System: "Manuscript"

### Visual Concept
The app embodies an editorial, book-like aesthetic — an elegant language textbook come to life. Warm cream backgrounds evoke paper, deep teal interactive elements feel like ink annotations, and rich gold accents celebrate achievement. Red teacher-style underlines mark errors. The experience feels crafted and personal, not generated.

## 3. Core Modules

### A. Dictation (Listen & Type)
1. App plays a sentence via TTS. A "Slow" button replays at 0.6x speed.
2. User types what they hear.
3. Feedback: a `WordDiff` component renders both sentences word-by-word — green for exact match, yellow for close (Levenshtein ≤ 2), red for wrong/missing. Partial credit via Levenshtein distance.

### B. Spelling Bee (Hear & Spell)
1. App plays a single word via TTS.
2. User types the word. No text hints visible (definition shown only after submission).
3. Scoring: exact match = full points, close spelling = partial credit.

### C. Speaking (Read & Speak)
1. App displays a sentence on screen.
2. User taps "Start Recording", reads aloud, taps "Stop".
3. Browser STT transcribes speech.
4. Feedback: word-level comparison between transcript and target. Levenshtein distance for fuzzy matching (accounts for STT inaccuracy).

## 4. Database Schema

### `profiles`
* `id` (UUID, PK) — references `auth.users`.
* `email` (Text).
* `full_name` (Text).
* `avatar_url` (Text).
* `updated_at` (Timestamp).

### `missions`
* `id` (UUID, PK).
* `day_number` (Int, unique) — sequential day identifier.
* `title` (Text) — e.g., "Effective Email Communication".
* `description` (Text).
* `content` (JSONB) — structured exercises:
    ```json
    {
      "dictation": [
        {"text": "Please schedule the meeting.", "hint": "Business"}
      ],
      "spelling": [
        {"word": "schedule", "definition": "To arrange a time for an event."}
      ],
      "speaking": [
        {"text": "I will send the agenda by noon."}
      ]
    }
    ```

### `user_progress`
* `id` (UUID, PK).
* `user_id` (FK → `auth.users`).
* `mission_id` (FK → `missions`).
* `status` (Text: `locked` | `open` | `completed`).
* `score` (Int, 0-100).
* `completed_at` (Timestamp).
* Unique constraint on `(user_id, mission_id)`.

## 5. Progression & Scoring

### Scoring
Each exercise scores 0-100 points. The mission score is the average across all exercises.

* **Dictation:** Word-level comparison. Each correct word earns proportional points. Close misspellings (Levenshtein distance ≤ 2) earn partial credit.
* **Spelling:** Exact match = 100. Levenshtein distance 1 = 75. Distance 2 = 50. Otherwise 0.
* **Speaking:** Word-level overlap between STT transcript and target. Generous scoring to account for STT imprecision.

### Progression Rules
* When a user first loads the dashboard, progress rows are initialized for all missions (Day 1 = `open`, rest = `locked`).
* Completing a mission (all exercises finished) unlocks the next day.
* No minimum score required to progress — the goal is practice, not gatekeeping.
* Users can revisit and redo any completed mission. On redo, the higher score is kept.

### Streaks
* Track consecutive days with at least one completed mission.
* Display current streak on the dashboard.

## 6. UI/UX Flow

### Dashboard
1. **Header:** App title, current streak, total points.
2. **Today's Mission card:** Prominent card showing the next open mission with "Start" button.
3. **Mission list:** All missions with status indicators (locked/open/completed) and scores.

### Mission Runner (Step-by-Step Wizard)
1. **Progress bar** at top showing current step (e.g., "Spelling 2/5").
2. **Step 1 — Warm-up (Spelling):** 5 words, one at a time.
3. **Step 2 — Focus (Dictation):** 3 sentences, one at a time.
4. **Step 3 — Action (Speaking):** 2 sentences, one at a time.
5. **Step 4 — Summary:** Total score, breakdown by module, "Next" button to return to dashboard.

### Per-Exercise Flow
1. User completes exercise → sees feedback (correct answer, their answer, score).
2. User clicks "Next" to advance (no auto-advance).
3. Score is accumulated and shown in the header.

## 7. Browser Support & Limitations

| Feature | Chrome/Edge | Safari | Firefox |
|---------|-------------|--------|---------|
| TTS | Full | Full | Full |
| STT | Full (online) | Full (on-device) | Not supported |

* **Primary target:** Chrome/Edge on desktop and Android; Safari on iOS.
* **Fallback for unsupported browsers:** Speaking module shows a "not supported" message with a suggestion to use Chrome. Dictation and Spelling still work.
* **Mobile considerations:** Touch-friendly buttons (min 44px tap targets). No hover-dependent interactions.

## 8. PWA Scope

* **Installable:** App shell cached via service worker. Users can add to home screen.
* **Offline-capable (partial):** Dashboard and mission content can be cached after first load. TTS works offline (browser-native). STT may require network on Chrome.
* **Not offline:** Authentication, progress syncing.

## 9. Deployment

* **Vercel:** Connect GitHub repo. Auto-deploy main branch.
* **Supabase:** Hosted PostgreSQL. Free tier covers this use case.
